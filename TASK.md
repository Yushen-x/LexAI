### 1.1 接入得理开放平台API

**这是什么：** 得理在腾讯元器上提供了法条和案例的语义检索插件。通过这个接口，先检索出相关法条和案例，再把结果作为上下文塞进大模型的Prompt里，让模型基于真实法律数据作答，而不是凭空生成（这就是RAG——检索增强生成）。

**具体要做什么：**
- [ ] 在腾讯元器平台注册账号，找到得理插件，申请接口权限，获取API Key
- [ ] 新建 `DeliLegalSearchClient.java`，封装法规检索接口：传入问题描述，返回相关法条列表（法条编号、条文内容、所属法规名称）
- [ ] 新建 `DeliCaseSearchClient.java`，封装案例检索接口：传入案情描述，返回相似案例列表（案号、裁判要旨、判决结果）
- [ ] 在调用大模型前先调用检索接口，将结果格式化后拼入Prompt上下文（"以下是相关法条/案例，请基于此作答..."）
- [ ] 处理检索接口超时或无结果的情况，保证检索失败时大模型仍能正常响应

### 1.2 RAG 检索增强生成

**这是什么：** 除了得理提供的外部API，还需要搭建一套本地知识库，存放我们自己整理的法律材料（常见合同模板、典型判例摘要、常用法条解读等）。这些材料经过向量化后存入向量数据库，用户提问时先在本地知识库里检索相关片段，再连同得理API的检索结果一起注入Prompt，让模型输出更准确、更有针对性。

RAG的核心流程：用户提问 → 问题向量化 → 在向量库里找最相似的文档片段 → 把片段 + 得理检索结果 + 用户问题一起发给大模型 → 模型基于真实材料作答。

**具体要做什么：**
- [ ] **整理知识库材料**：收集并整理以下内容，存为文本文件（每份500-1000字为宜，来源要可靠）：
  - 常见法律场景的核心法条解读（劳动法、合同法、侵权责任法各5-10条核心条款）
  - 3-5套常见合同模板（服务合同、采购合同、劳动合同、保密协议、租赁合同）
  - 10-20个典型判例摘要（涵盖劳动纠纷、合同违约、侵权等场景）
- [ ] **文档切分（Chunking）**：将收集的材料切分成合适大小的文本块（建议每块200-500字，相邻块之间保留50字重叠，避免语义截断），为每个文本块记录来源信息（文件名、章节）
- [ ] **向量化与存储**：选用腾讯云向量数据库或本地轻量方案（如基于文件的简单向量索引），将每个文本块调用 embedding 模型生成向量并存储
- [ ] **检索接口实现**：新建 `LocalKnowledgeSearchClient.java`，传入用户问题，返回余弦相似度最高的Top-3文档片段（含来源信息）
- [ ] **整合进Prompt**：在1.1的得理API检索结果基础上，额外加入本地知识库检索到的片段，按优先级排序后拼入Prompt上下文（得理案例/法条 > 本地模板/解读）
- [ ] 评估RAG效果：对比有无RAG时的输出差异，确认检索结果对输出质量有正向提升
