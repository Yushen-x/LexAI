# LexAI Demo 前 Bug 与数据方案清单

更新时间：2026-04-22

本文档分两部分：

- 第一部分：当前系统的数据存储方式与部署建议
- 第二部分：Demo 前建议修复的 Bug 清单（包含大 Bug 和小 Bug）

目标不是一次性做成完整产品，而是优先保证：

- 核心链路正确
- Demo 结果可信
- 系统运行稳定
- 页面和文案不自相矛盾

## 一、当前系统的数据是怎么存的

### 1.1 当前实际存储方式

当前项目后端使用的是 **H2 文件型数据库**。

从配置文件看：

- 数据库地址：`jdbc:h2:file:./data/lexai-db`
- 配置文件：`backend/src/main/resources/application.yml`

这意味着：

- 数据不是只存在内存里
- 数据会落到后端目录下的本地文件中
- 当前实际数据库文件在：
  - `backend/data/lexai-db.mv.db`
  - `backend/data/lexai-db.trace.db`

也就是说，只要你后端运行在某台机器上，H2 就会把数据保存在那台机器的磁盘里。

### 1.2 当前这种方式适不适合 Demo

适合。

如果只是做比赛初赛 Demo，这种方式有几个优点：

- 配置简单
- 启动快
- 不需要额外安装复杂数据库
- 对单机演示非常友好

所以：

> 对于“初赛演示环境”，完全可以把数据先存在部署后端的那台服务器上。

### 1.3 如果要部署，是不是需要服务器

是的。

如果你们要让评委访问一个真实可打开的 Demo，最简单的方案仍然需要一台服务器，常见做法是：

1. 一台云服务器运行后端 Spring Boot
2. 同一台服务器部署前端静态页面
3. 数据也先存在这台服务器上

最简单可行的部署结构：

- 前端：构建后的静态文件
- 后端：Spring Boot jar
- 数据库：先用 H2 文件库，直接放在服务器磁盘上

### 1.4 数据能不能直接存在服务器上

可以，而且对初赛来说这是最省事的方案。

你们可以这样理解：

- **前端** 放在服务器上
- **后端接口** 放在服务器上
- **数据库文件** 也放在服务器上

这对单机 Demo 完全可行。

### 1.5 但 H2 适不适合长期/正式使用

不太适合。

H2 更适合：

- 本地开发
- 单机演示
- 小规模测试

如果你们后面要：

- 多人同时访问
- 长期部署
- 保证更稳定的数据持久化
- 做更正式的线上展示

更建议改成：

- MySQL
- PostgreSQL

而且数据库也可以继续放在同一台服务器上，不一定非得单独买一台数据库服务器。

### 1.6 对你们现在最推荐的部署策略

按“初赛 Demo 优先”的原则，推荐分两档：

#### 方案 A：最省事 Demo 方案

- 1 台服务器
- 前端静态文件部署在这台服务器
- Spring Boot 后端部署在这台服务器
- H2 文件数据库也放在这台服务器

适用场景：

- 初赛提交
- 录屏演示
- 少量评委访问

#### 方案 B：更稳一点的 Demo 方案

- 1 台服务器
- 前端静态文件部署在服务器
- Spring Boot 后端部署在服务器
- 数据库改成 MySQL，也装在同一台服务器

适用场景：

- 希望更像正式系统
- 希望减少 H2 带来的演示风险
- 需要更稳定的数据一致性

### 1.7 当前建议

对于你们现在这个阶段，我的建议是：

> 如果时间紧，先不急着改数据库技术栈，先把核心 Bug 修掉，保证 H2 单机 Demo 稳定。

因为对初赛来说，评委更关心的是：

- 你们能不能稳定演示
- 链路是否完整
- 结果是否专业

而不是你们是不是已经上了生产级数据库架构。

## 三、P0：必须优先修复

### Bug 1：合同编号生成存在重复风险

- 优先级：`P0`
- 类型：后端逻辑 / 数据正确性
- 位置：`backend/src/main/java/com/lexai/backend/application/service/ContractService.java`
- 当前逻辑：
  - 编号通过 `countByDeletedFalse() + 1` 生成
- 问题：
  - 软删除后重新创建合同，可能复用旧编号
  - 并发创建时，两个请求可能拿到相同编号
  - 会触发数据库唯一键冲突
- 影响：
  - 新建合同失败
  - Demo 中“保存草稿 / 新建合同”可能直接报错
- 建议修法：
  - 不要依赖 count 生成编号
  - 改为基于数据库最大编号生成，或单独维护序列
  - Demo 阶段最简单方案：查询当年最大流水号，再 `+1`

### Bug 2：任务编号生成存在重复风险

- 优先级：`P0`
- 类型：后端逻辑 / 数据正确性
- 位置：`backend/src/main/java/com/lexai/backend/application/service/TaskService.java`
- 当前逻辑：
  - 编号通过 `taskRepository.count() + 1` 生成
- 问题：
  - 并发创建时可能撞号
  - 后续数据重置/迁移时不稳定
- 影响：
  - 自动生成待办时可能失败
  - 会影响 Demo 闭环
- 建议修法：
  - 和合同编号一样，改成基于最大流水号或独立序列生成

### Bug 3：“存草稿”实际上没有保存合同正文

- 优先级：`P0`
- 类型：业务逻辑 / 语义错误
- 位置：
  - 前端：`frontend/src/modules/contract/views/ContractDraftView.vue`
  - 后端 DTO：`backend/src/main/java/com/lexai/backend/application/dto/contract/CreateContractRequest.java`
  - 实体：`backend/src/main/java/com/lexai/backend/persistence/entity/ContractEntity.java`
- 当前逻辑：
  - 前端点击“存草稿”后，仅保存合同基础信息
  - 没有保存正文内容
- 问题：
  - 语义上它不是“草稿”，而只是“合同台账记录”
  - 用户之后无法真正基于已保存的合同正文继续编辑
- 影响：
  - 如果评委追问“草稿以后怎么继续打开编辑”，当前逻辑说不通
- 建议修法：
  - 两种选法二选一：
  - 方案 A：真正支持草稿，把正文 `content` 存进后端
  - 方案 B：不改数据结构，但把按钮文案从“存草稿”改成“加入台账”或“保存记录”

### Bug 4：起草页跳审查页使用 `sessionStorage`，但数据不会自动清理

- 优先级：`P0`
- 类型：前端状态污染 / Demo 稳定性
- 位置：
  - 写入：`frontend/src/modules/contract/views/ContractDraftView.vue`
  - 读取：`frontend/src/modules/contract-review/views/ContractReviewView.vue`
- 当前逻辑：
  - 起草页把内容写入 `sessionStorage`
  - 审查页读取后不清理
- 问题：
  - 下一次进入合同审查页时，旧数据可能还在
- 影响：
  - 很容易在 Demo 时串场
  - 让人误以为系统逻辑混乱
- 建议修法：
  - 审查页在成功读取后立即清除对应 `sessionStorage`
  - 或在手动重置时清理掉

### Bug 5：主链路里的“保存”和“流转”缺少真实业务关联 ID

- 优先级：`P0`
- 类型：业务闭环可信度
- 位置：
  - `LegalWorkspaceServiceImpl.java`
  - `TaskService.java`
- 当前逻辑：
  - 自动生成待办时使用的是随机 `relatedId`
- 问题：
  - 待办记录和真实业务对象之间关联较弱
  - 很难解释“这条待办到底对应哪份合同/哪次操作”
- 影响：
  - Demo 看上去有闭环，但追问下去会显得关系不强
- 建议修法：
  - 合同相关流程尽量把真实合同 ID 或合同编号写入 `relatedId`
  - 至少在 Demo 主链路里做到“能解释得通”

## 四、P1：建议尽快修复

### Bug 6：首页把 AI 状态写死成“Mock 网关”

- 优先级：`P1`
- 类型：前端展示 / 口径冲突
- 位置：`frontend/src/modules/dashboard/views/DashboardView.vue`
- 当前逻辑：
  - 系统状态里写死为 `Mock 网关`
- 问题：
  - 与当前“真实 AI + 检索增强 + 兜底”的演示口径冲突
- 影响：
  - 评委会直接看到系统自己写着 Mock
- 建议修法：
  - 改成中性文案，例如：
    - `AI 网关已接入`
    - `检索增强已启用`
    - `智能推理可用`

### Bug 7：草稿校验逻辑过严，不符合“草稿”语义

- 优先级：`P1`
- 类型：产品逻辑
- 位置：`frontend/src/modules/contract/views/ContractDraftView.vue`
- 当前逻辑：
  - “存草稿”复用了正式生成合同的校验逻辑
- 问题：
  - 草稿通常应允许不完整输入
- 影响：
  - 用户体验不自然
  - Demo 里“草稿能力”不够真实
- 建议修法：
  - 如果保留“草稿”概念，应放宽校验
  - 如果不想放宽，就改按钮文案，避免误导

### Bug 8：异常提示语言中英混杂

- 优先级：`P1`
- 类型：用户体验 / 专业感
- 位置：`backend/src/main/java/com/lexai/backend/common/exception/GlobalExceptionHandler.java`
- 当前逻辑：
  - 有些错误提示已经变成英文
- 问题：
  - 前端整体是中文界面，错误提示英文会显得很割裂
- 影响：
  - 降低 Demo 的统一感和专业感
- 建议修法：
  - 统一改回中文用户提示
  - 日志保留英文或中文都可以，但用户提示建议全中文

### Bug 9：合同审查页预填后，页面没有明显提示“来源于起草结果”

- 优先级：`P1`
- 类型：交互解释性
- 位置：`frontend/src/modules/contract-review/views/ContractReviewView.vue`
- 当前逻辑：
  - 数据会自动预填，但用户看不出这是从起草页带来的
- 问题：
  - 对演示链路来说少了一个“系统联动”的可视化确认
- 影响：
  - 链路存在，但不够显眼
- 建议修法：
  - 增加一个小提示条，例如：
    - `已从合同起草页自动带入内容`

### Bug 10：合同详情和任务详情目前仍是 `alert` 弹出

- 优先级：`P1`
- 类型：交互表现
- 位置：
  - `frontend/src/modules/contract/views/ContractListView.vue`
  - `frontend/src/modules/workflow/views/WorkflowPendingView.vue`
- 问题：
  - 技术上能用，但很像临时开发态
- 影响：
  - 在 Demo 中会显得完成度稍低
- 建议修法：
  - 不一定要做完整详情页
  - 最少可以改成简易弹窗或侧边面板

## 七、最后结论

你们当前最重要的不是“继续加很多功能”，而是先保证这套系统在 Demo 语境下满足四个标准：

- **逻辑正确**
- **数据可信**
- **链路完整**
- **展示稳定**

一句话判断：

> 先把“系统是对的”做扎实，再去追求“系统看起来很强”。

对初赛来说，真正最重要的是：

- 起草能生成
- 审查能给出结构化风险
- 台账能沉淀记录
- 待办能形成闭环
- 全程不报错、不串数据、不自相矛盾
