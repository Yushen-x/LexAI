# LexAI 初赛优先级任务清单

## 目标

当前目标不是继续堆新功能，而是把现有能力打磨成一个：

- 能稳定演示
- 看起来专业且完整
- 符合初赛评分口径
- 对外材料统一、不露馅

建议只围绕一条主链路做强展示：

1. 工作台
2. 合同起草
3. 提交审查
4. 风险报告
5. 合同台账
6. 待办流转

再补一个短亮点：

- 法律咨询
或
- 案件分析

---

## P0：必须最先完成

这些不修，Demo 容易翻车，或者评委一追问就失分。

### 1. 修主链路真闭环

- [ ] 合同起草页“提交审查”不能只写 `sessionStorage`
- [ ] 提交审查前先确保合同已落库
- [ ] 提交审查时把合同状态切成 `UNDER_REVIEW`
- [ ] 审查结果要和这份合同绑定，而不是一次性页面结果
- [ ] 待办任务的 `relatedId` 尽量改成真实合同 ID 或可追踪业务主键

交付标准：

- 从起草页提交后，在台账里能看到同一份合同状态已变为“审查中”
- 待办里能找到对应任务
- 详情里能看出这是同一份合同的连续流转

### 2. 修合同审查失败时保留旧结果的问题

- [ ] 点击重新审查前先清空旧报告
- [ ] 审查失败时页面不要残留上一份成功结果
- [ ] 给出明确错误提示，但不要误导用户认为当前结果有效

交付标准：

- 失败时右侧不再显示旧报告
- 录屏时不会出现“失败了但报告还在”的尴尬画面

### 3. 限制无效待办生成

- [ ] 合同起草页侧边问答不要自动生成正式待办
- [ ] Agent 改稿不要自动生成正式待办
- [ ] 只有真正业务动作才生成任务，例如正式咨询、正式审查、正式起草提交

交付标准：

- 工作台待办数不会因为聊天或试验性输入无意义膨胀
- 数据看起来更真实、更可信

### 4. 修台账和待办页的状态同步问题

- [ ] 台账状态修改后，详情区和列表保持同步
- [ ] 待办状态修改后，详情区和列表保持同步
- [ ] 选中详情项在刷新后尽量保留或自动更新

交付标准：

- 演示时不会出现“列表改了、详情没改”的割裂感

### 5. 统一对外口径，去掉容易露馅的话术

- [ ] 首页“数据存储：MySQL 持久化”“AI 网关已接入”等静态说明改成更稳妥表述
- [ ] 不再对外写“多智能体协同推理”，除非你们真做了可展示的多智能体
- [ ] 更新 README 和旧文档里“Mock/H2/按钮未接线/无后端/8080”等过时内容
- [ ] 元器相关文档改成“已完成配置方案设计，代码待正式接入”这类准确口径，避免写成“已接入”但代码里没有

推荐对外一句话：

> LexAI 是一个基于腾讯混元大模型、得理法规/类案检索和本地知识库增强的智慧法律工作台，重点解决合同起草、合同审查、台账管理与任务流转场景。

### 6. 出一份统一可交初赛的成稿材料

- [ ] 项目简介
- [ ] 需求分析
- [ ] 核心功能说明
- [ ] 使用场景
- [ ] AI 工具使用说明
- [ ] Prompt / 工作流说明

交付标准：

- 不再依赖 README、架构文档、成员说明东拼西凑
- 有一份可直接给评委看的主文档

## P1：高性价比加分项

这些做完会明显提升“看起来很厉害”的感觉，也是初赛很值的分数来源。

### 1. 固定 Demo 剧本和讲稿

- [ ] 固定 1 条主链路，不平均展示所有功能
- [ ] 每一步对应一句讲解词
- [ ] 准备开场、过渡、结尾三段固定话术

建议脚本：

1. 进入工作台，看总览和待办
2. 打开合同起草，生成一份采购/技术服务合同
3. 一键提交审查，进入风控报告
4. 展示高风险条款和修改建议
5. 回到合同台账，查看同一合同状态变化
6. 打开待办，完成一次状态流转
7. 最后补一个法律咨询或案件分析亮点

### 2. 准备 3 到 5 组稳定演示样例

- [ ] 合同起草样例 2 组
- [ ] 合同审查样例 2 组
- [ ] 法律咨询或案件分析样例 1 组

样例要求：

- 一定能稳定出结果
- 一定容易解释
- 一定能体现专业性

### 3. 做一页系统架构图

- [ ] 前端 Vue
- [ ] 后端 Spring Boot
- [ ] 腾讯混元
- [ ] 得理开放平台 API
- [ ] 本地知识库
- [ ] 合同台账 / 待办任务 / 数据库

用途：

- PPT
- 项目简介
- Demo 视频开头或结尾

### 4. 做一页 AI 工作流图

- [ ] 用户输入
- [ ] 业务场景识别
- [ ] 得理法规/类案检索
- [ ] 本地知识库检索
- [ ] 腾讯混元生成
- [ ] 结构化结果返回
- [ ] 台账 / 待办 / 工作流流转

### 5. 补专业验证记录

- [ ] 找 1 名法学老师、法学生或法务同学试用
- [ ] 记录测试问题
- [ ] 记录输出是否准确、是否实用
- [ ] 记录意见和修改动作

建议最少做成一张表：

- 测试人身份
- 测试场景
- 评价
- 发现问题
- 你们的优化动作

### 6. 统一页面文案与状态命名

- [ ] 合同类型前后统一
- [ ] 状态名称统一
- [ ] 首页、台账、待办、审查页的文案风格统一

## 当前服务器连通性验证

已完成一次外网验证，结果如下：

- [x] `http://124.223.111.253/` 可访问
- [x] `http://124.223.111.253/api/actuator/health` 返回 `200`
- [x] 公网验证脚本 `docs/verify_public_workflow.ps1` 可跑通
- [x] 可成功新增合同、查询合同、调用法律咨询、合同起草、合同审查
- [x] 可成功新增待办任务

本次验证结果摘要：

- 合同总数：`9 -> 10`
- 新建合同成功，合同号：`LX-2026-011`
- 合同查询内容一致：`fetchedContentMatch = true`
- 法律咨询返回了分类、法律依据和建议
- 合同起草返回正文，正文长度：`2647`
- 合同审查返回摘要、风险项和缺失条款
- 任务数：`14 -> 17`

当前服务器层面结论：

- 你的服务器是通的
- Nginx 是通的
- 后端接口是通的
- 主链路 API 基本是通的

但也暴露了一个现状问题：

- 当前正式调用咨询 / 起草 / 审查后，都会自动新增待办
- 这会导致任务列表在 Demo 过程中增长很快，需要在 P0 里控制

## 推荐执行顺序

1. 先修主链路真闭环
2. 再修审查失败残留旧结果
3. 再限制无效待办生成
4. 再统一对外文案和材料口径
5. 再固定 Demo 剧本和样例
6. 最后补架构图、工作流图、专业验证和录屏

LexAI 现在已经不是“做不出来”的问题，而是“要把现有能力打磨成一个高分初赛 Demo”的问题。
