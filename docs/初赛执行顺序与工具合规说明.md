# 初赛执行顺序与工具合规说明

## 1. 现在用混元，是否不符合要求？

结论：

- **不建议现在因为这个问题推倒重做平台**
- **当前继续使用腾讯混元是可以接受的**
- **真正的风险不在“用了混元”，而在“你们怎么说明自己用了腾讯系工具”**

### 1.1 为什么不建议现在改平台

从你给我的赛题要求看，比赛强调的是：

- 基于腾讯系 AI 工具平台
- 合理使用腾讯系工具完成开发与展示

你们现在已经明确具备这些腾讯生态元素：

- 腾讯混元大模型能力
- CodeBuddy 作为开发过程工具的合理叙述空间
- 腾讯开悟比赛平台作为赛题获取与工具链使用入口

同时你们还用了：

- 得理开放平台 API

所以从方向上看，**不是明显不符合要求**。

### 1.2 真正要注意的点

如果你们在提交材料里只写：

> 我们调用了混元 API

那会有两个问题：

1. 评委看不出你们如何“组合使用腾讯系工具”
2. `AI工具运用深度` 这一项容易丢分

### 1.3 最稳妥的口径

建议对外这样表述：

- 大模型能力：腾讯混元
- 开发辅助：CodeBuddy / CodeBuddy Agents
- 比赛平台与工具链学习：腾讯开悟比赛平台
- 法律检索增强：得理开放平台 API
- 可补充说明：腾讯元宝用于 Prompt 打磨、需求整理、展示文案辅助

### 1.4 要不要现在切到元器？

结论：

- **不是必须**
- 如果你们现在系统已经稳定，不建议为初赛临时大改底层
- 但如果时间允许，**补一个轻量的元器工作流说明、截图、或一个简单元器智能体页**，会更保险

一句话：

> 现在最大的问题不是“必须不用混元”，而是“必须把腾讯系工具使用链路说清楚”。 

---

## 2. 初赛最优执行顺序

当前建议严格按下面顺序推进，不要分散火力：

### 第一步：先修影响 Demo 正确性的关键问题

先修这些，不然演示时容易翻车：

1. 提交审查是否真正落库
2. 合同状态是否真实流转
3. 审查失败时是否残留旧结果
4. 详情区与列表刷新是否同步

原则：

> 先保证系统是“对的”，再谈系统“好看”。

### 第二步：锁定 Demo 主链路

建议你们只打这一条主线：

1. 工作台
2. 合同起草
3. 合同审查
4. 合同台账
5. 待办流转

然后只补一个辅助亮点：

- 法律咨询
或
- 案件分析

不要平均展示所有功能。

### 第三步：固定演示输入与输出

每个演示模块准备一组“稳定样例”：

1. 合同起草样例
2. 合同审查样例
3. 法律咨询或案件分析样例

要求：

- 一定稳定出结果
- 一定看起来专业
- 一定容易讲解

### 第四步：确保线上环境稳定

这一阶段重点检查：

1. 前端公网访问是否正常
2. 后端接口是否稳定
3. MySQL 是否真实写库读库
4. 腾讯混元 / 得理 / 本地知识库配置是否正常
5. Demo 主链路是否连续跑通

### 第五步：补齐评分导向材料

这一阶段开始整理：

1. 项目简介
2. AI 工具使用说明
3. Prompt / 工作流说明
4. 系统架构图
5. AI 工作流图
6. 专业验证测试记录

### 第六步：录 Demo 视频

只有前五步完成后，才建议开始正式录屏。

录屏前要先做：

1. 全流程彩排
2. 确认话术
3. 确认页面和数据状态

---

## 3. 当前按优先级排序的任务清单

## P0：必须先做

1. 修“提交审查”的真实闭环
2. 修“审查失败保留旧结果”
3. 修“详情区与列表刷新不同步”
4. 固定 Demo 主链路
5. 固定演示样例输入
6. 跑通线上稳定环境

## P1：接着做

1. 写 `AI工具使用说明（工作流 + Prompt）`
2. 写 `初赛版项目简介`
3. 补 `专业验证测试记录`
4. 做 `系统架构图`
5. 做 `AI 工作流图`

## P2：最后做

1. 录 Demo 视频
2. 彩排答辩口径
3. 统一系统文案与展示话术
4. 做补充材料整理

---

## 4. 你们现在最稳的策略

如果目标是初赛高分而不是长期产品化，那么最优策略是：

> 不要为了“平台更纯正”去临时大改底层；  
> 要优先把“系统正确性、Demo 稳定性、腾讯系工具使用说明、专业验证材料”补齐。

## 5. 一句话结论

### 关于混元

**现在不用急着换掉。**

### 关于优先级

**先修关键逻辑，再锁定 Demo，再整理材料，最后录视频。**
