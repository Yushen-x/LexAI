# 初赛差距清单

## 当前判断

LexAI 目前已经具备初赛可演示的核心形态：

- 有明确的智慧法律应用场景
- 有可运行的前后端系统
- 有公网 Demo
- 有法律咨询、案件分析、合同起草、合同审查、合同台账、待办任务等核心模块
- 已完成 MySQL 持久化和基础部署

但从“能演示”到“能拿高分”之间，还存在几类明显差距：

- 业务闭环还不够完全严谨
- 腾讯系工具使用证据链不足
- 专业验证材料不足
- 提交材料尚未完全按评分标准组织

## 二、当前主要差距

## A. 业务与 Demo 闭环差距

### A1. 合同起草到合同审查的业务闭环还不够严谨

当前“提交审查”主要是将内容写入 `sessionStorage` 后跳转到审查页。  
这意味着页面展示上已经像一个闭环，但底层还不够完全体现：

- 当前最新正文是否已持久化
- 当前合同状态是否切换为 `UNDER_REVIEW`
- 审查结果与台账记录是否是同一份合同的连续流转

这是最值得优先修复的演示级逻辑问题。

### A2. 审查页失败时可能残留上一次成功结果

如果用户本次审查失败，而页面仍保留旧报告，会直接影响 Demo 可信度。

### A3. 台账/待办详情与列表刷新联动还不够稳

当前详情卡片已经替代了 `alert`，这是明显进步；  
但状态修改后，详情区与列表重新加载之间仍可能出现短暂不同步。

### A4. 待办与真实业务对象的关联仍偏弱

当前待办 `relatedId` 仍偏“流程标识”，不是强业务主键。  
因此待办虽然能展示，但“点击待办直达业务对象”的平台闭环感还不够强。

## B. 赛题要求匹配差距

### B1. 腾讯系工具使用说明不够成体系

这是当前最容易影响评分的部分之一。

赛题强调：

- 腾讯开悟比赛平台
- CodeBuddy
- 腾讯元器
- 腾讯元宝
- 小理AI
- 得理开放平台 API

当前项目在代码和配置层面可以明显看出：

- 已接入腾讯大模型
- 已接入得理法规/案例检索
- 已有 Prompt 与检索增强逻辑

但仍然缺少一份完整的评审口径说明，明确回答：

- 每个工具具体承担什么作用
- 工具之间如何协同
- Prompt 如何设计
- 为什么这样设计工作流

### B2. 专业验证测试不足

当前更像工程验证充分，但法律专业验证证据链不足。

仍缺：

- 法学老师 / 法学生 / 法务人员测试记录
- 输出准确性评价
- 反馈意见与改进说明

### B3. 提交材料尚未完全围绕评分标准组织

现有内容分散在：

- README
- ProjectCharter
- ARCHITECTURE
- 成员说明
- API 配置文档

但初赛更需要一份统一口径的材料，严格覆盖：

- 项目基础信息
- 项目需求分析
- 关键功能说明
- 使用场景
- AI 工具使用说明（工作流 + Prompt）

## C. 演示呈现差距

1. 工作台
2. 合同起草
3. 提交审查
4. 风险识别
5. 合同台账

或

目前功能具备，但演示叙事仍需进一步固化。

### C3. 仍缺视频、流程图、工具链说明图

这些属于初赛很重要的“感知分”来源。

## 六、当前实际使用的 AI API / AI 能力情况

## 1. 代码默认配置

后端主配置中：

- `LEXAI_AI_MODE` 默认是 `tencent`
- 腾讯大模型 endpoint 配置项为 `TENCENT_LLM_ENDPOINT`
- 腾讯模型名配置项为 `TENCENT_LLM_MODEL`

对应位置：

- `backend/src/main/resources/application.yml`
- `backend/.env.example`

## 2. 当前线上实际运行配置

已核对服务器上的 `lexai.env`，当前线上实际运行是：

- AI 模式：`tencent`
- 腾讯大模型接口：`https://api.hunyuan.cloud.tencent.com/v1/chat/completions`
- 腾讯模型：`hunyuan-lite`
- 得理法规/案例检索：已配置
- 本地知识库路径：已配置

出于安全原因，这里不记录具体密钥值。

## 3. 也就是说，你现在实际调用的是

### 主大模型

- 腾讯混元的 OpenAI 兼容 `Chat Completions` 接口
- 当前模型为：`hunyuan-lite`

### 检索增强

- 得理开放平台 API
  - 法规检索
  - 类案检索

### 本地增强

- 本地知识库检索

## 4. 当前 AI 调用逻辑不是“只调一个模型”，而是

更准确地说，你们现在是：

> 腾讯混元大模型 + 得理法规/类案检索 + 本地知识库检索 的组合式法律 AI 工作流

## 5. 当前还需要注意的一点

虽然线上配置已启用腾讯模式，且相关凭据已配置，  
但代码设计上仍然保留了兜底逻辑：

- 若腾讯调用失败
- 若返回内容解析失败

则系统会退回：

- 模板生成
- 规则化分析
- 检索增强的保底输出

这对 Demo 稳定性是好事，但在对外表达时要注意口径：

- 可以说“已接入腾讯混元 + 得理检索增强”
- 不建议说成“所有结果都完全由腾讯模型稳定生成”

更稳妥的表达是：

> 系统已接入腾讯混元大模型，并融合得理法规/案例检索与本地知识库增强，当前具备多层兜底机制，以保证 Demo 阶段的可用性与稳定性。
